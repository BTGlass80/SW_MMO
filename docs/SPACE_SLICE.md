# Space Slice

The first space slice is a 2.5D tactical overlay for the Mos Eisley approach, toggled with `M`.

Current scope:

- flat tactical plane
- curated Clone Wars Mos Eisley approach contacts
- WEG-style range-band placeholders
- sensor sweeps with deterministic D6 rolls against contact range difficulty
- sensor sweep margins classify contacts as missed, faint, partial, solid, or clear tracks
- the best successful sensor confidence is persisted per contact and shown on status-model formatted revealed contact labels
- selected-contact identification scans use the current sensor track to resolve curated transponder, affiliation, and threat profile data
- selected-contact communications hails resolve data-driven responses, persist responsive/unresponsive comms disposition labels, can data-drive a successful-hail delay to the selected contact's weapon-solution clock, and can data-drive failed-hail pressure that advances an armed contact's lock with ready-lock callouts when pressure completes the firing solution, with comms lock context preserved on contact labels/details
- gunnery events annotate the target's current sensor-track confidence as a model-formatted targeting context and apply confidence-based difficulty modifiers to hidden or sensor-required targets
- contacts can start unresolved and reveal after successful sensor sweeps
- revealed contacts persist after later weaker or failed sensor sweeps, while each model-formatted sweep readout still reports successful tracks, newly revealed contacts, current known/hidden contact totals, and consumed sensor assists
- first seeded gunnery drill against the default unresolved contact, with a status-model formatted action readout
- hostile contacts can opt into seeded counterfire after the player's gunnery exchange, with return-fire results formatted by the compact status model
- ship shields add to hull soak for covered non-ion hits
- starship-shields station action can reroute listed shield dice across requested arcs with a status-model formatted action readout
- rotating crew-station assist actions can bank a one-shot bonus for the next matching piloting, sensors, communications, gunnery, shields, or repair action, include a commander tactical-coordination assist, normalize helm/piloting target aliases such as evasive or break-lock support to maneuver, normalize fire-control target aliases such as target lock or weapon solution to gunnery, normalize shield-operation aliases such as deflectors or angle shields to shields, normalize sensor-family target aliases such as identification or targeting solution to sensors, normalize communications-family target aliases such as hail contact or transponder ping to communications, normalize engineering-family aliases such as field repair or patching to repair, normalize navigation-family aliases such as route plot or jump calculation to astrogation, use a status-model formatted action readout, call out consumed assists and requested aliases in later action readouts, and report through the status model when a new assist replaces an older banked assist for the same action, preserving old/new requested target aliases in that replacement text
- navigator astrogation plotting rolls the ship's astrogation/nav pool, consumes a matching navigator assist, applies active hyperdrive calculation and astrogation difficulty penalties, clears those nav penalties on success, and fails without clearing state if the hyperdrive is disabled
- ion-tagged weapons bypass shields in the first gunnery model
- gunnery can derive the incoming shield arc from attacker/target positions and target heading
- piloting maneuver actions roll piloting plus maneuverability, update heading/position, record WEG movement failures on misses, and use a status-model formatted action readout
- maneuver actions can carry data-driven circular hazards that increase difficulty when the intended path crosses them
- failed piloting maneuvers can resolve WEG speed-based collision damage when the maneuver data says an obstacle is present
- non-ion gunnery hits are interpreted through the WEG starship damage bands and reported through the status-model gunnery action formatter
- ion gunnery hits are interpreted as escalating controls ionized results
- light/heavy/severe starship damage rolls deterministic WEG system-effect entries
- gunnery returns a lightweight persistent target `condition` block for shield loss, ionization, hull severity, system effects, and destruction
- later gunnery exchanges apply shield loss and controls-ionized penalties to shields, fire control, weapon damage, and defensive maneuver pools
- specific system effects now feed later gunnery: maneuverability loss reduces defensive maneuver pools, disabled drives remove defensive maneuver, disabled weapons prevent attacks, and lost Move/dead-in-space/destruction flags persist
- controls-ionized effects count down across ship-condition rounds
- damage-control repair actions use WEG repair difficulty bands, carry field-time and dockyard-cost metadata, can clear one targeted damaged system, and use a status-model formatted action readout
- damage-control repair events carry before/after ship condition summaries plus consumed crew-assist summaries for server audit trails and readable repair-change overlays
- live damage-control actions prioritize the player ship's repairable condition before falling back to the drill target
- ship condition summaries convert persistent damage state into readable status, repairable-system lists, and crew-wound counts for overlays; shared action readouts now use the compact status-model condition formatter
- passenger/gunner damage events use the WEG character-scale passenger damage chart for light, heavy, severe, and destroyed ship results
- named ship crew can roll soak against passenger/gunner damage and persist WEG wound results in ship condition
- shared WEG scale helpers for future ship, walker, vehicle, and station combat
- crew station list for active ship roles, now including commander support
- data-driven contacts from `data/space_tactical_slice.json`
- keyboard contact cycling can select next/previous contacts without relying on precise mouse clicks
- the selected-contact side-panel readout summarizes track quality, ID, comms disposition, range-aware weapon-solution lock pressure, WEG scale posture, defensive posture, hull/shield soak posture, weapon posture, crew wound posture, systems posture, bridge cue with action hotkeys, movement posture, hostile-fire readiness, counterfire posture, destroyed-weapon-aware and custom data-preserving field-repair options/routing with visible data-driven repair difficulty labels, sensor-targeting difficulty, and condition without revealing hidden-contact identity too early
- click and keyboard target selection update the scan readout and append a compact target-selection entry to the recent-action log
- manual traffic stepping advances one server-style accounting tick with `;` or the side-panel `Step` button, while `T` remains pause/resume for the live traffic clock
- bridge-mode presentation uses a smoke-tested mode model for the header status line and state-aware action button labels, with smoke-tested wrapped layout slots for header and traffic text so selected target, known tracks, scan round, traffic state, live tick count, current maneuver difficulty/named-hazard preview, and selected-contact next-action cue stay visible as ship state rather than ground-scene HUD; the maneuver action button also mirrors the current route difficulty with a compact hazard marker/count and tooltip, selected-contact bridge cues highlight and tint their matching action buttons through structured cue metadata, the bridge header and highlighted action buttons now color the active cue by smoke-tested urgency level, and header/button tooltip wording distinguishes alert, threat, repair, and next-action guidance before the player commits
- the bridge crew/station strip uses a smoke-tested station-strip model to show crew assignments including commander, persistent crew wounds by station, and banked assist targets/bonus dice with requested target aliases and the station round when the assist was banked, making rotating station support and wounded crew roles visible before the assisted action consumes it
- persistent crew wounds now feed WEG station penalties for sensor sweeps, contact identification, piloting maneuvers, gunnery attacks and defensive maneuver pools, shield reroutes, damage control, astrogation plots, communications hails, and crew-station assists; wounded/stunned crew lose 1D, incapacitated crew zero the matching station pool, and smoke-tested action readouts call out the wound penalty that affected the roll
- a capped recent-action log preserves the last few player-triggered space actions while leaving automatic traffic ticks out of the log; when the player follows a highlighted bridge cue, the log tags that entry with compact Alert/Threat/Repair/Status/Next context from the same smoke-tested cue urgency model, then consumes the cue tag so later target or hazard selections do not inherit stale urgency
- contact movement profiles can advance selected contacts across the tactical plane, with destroyed, drives-disabled, and control-locked contacts reporting explicit movement-block reasons
- contact movement profiles can optionally track the player ship with a capped turn rate
- player-tracking contacts can hold a configured engagement range instead of closing forever
- selected contacts now show model-formatted WEG scale posture, keeping cross-scale attack, dodge, damage, and soak context visible before a gunnery roll
- selected contacts now show model-formatted defensive posture from their WEG defensive maneuver pool plus controls-ionized, maneuverability-loss, drives-offline, controls-dead, or destroyed condition state
- selected contacts now show model-formatted soak posture from hull, shield, shield-arc, shield-loss, and controls-ionized state, matching the same resistance inputs used by WEG starship damage without resolving a damage roll
- selected contacts now show model-formatted weapon posture from gunnery pool, fire control, weapon damage, controls-ionized, weapons-offline, and destroyed state, matching the same offensive inputs used by WEG space gunnery without resolving an attack roll
- selected contacts now show model-formatted crew wound posture from persisted passenger/gunner damage packets, including stunned station crew that still matter for WEG station penalties
- selected contacts now show model-formatted systems posture from Move loss, hyperdrive disabled/calculation state, astrogation difficulty penalties, generator overload, structural damage, and destroyed state, keeping non-weapon/non-defense ship condition visible without resolving repair
- selected contacts now show a model-formatted bridge cue with action hotkeys derived from existing sensor, identification, comms, weapon-solution, sensor-targeting, repair, and destruction state; that cue is echoed in the bridge header, and the bridge action row highlights/tints matching hotkey buttons from structured cue metadata, making the next useful bridge action easier to spot without adding a new rules gate
- the bridge cue falls back to local player-ship damage control when the selected contact has no higher-priority cue and the local ship has field-repairable damage, matching the live damage-control action's player-first repair routing
- bridge cue priority is smoke-tested in the status model: destroyed player ship overrides all, selected-contact threats/actions outrank routine local repairs, and local field repair fills otherwise-idle bridge guidance
- bridge cue urgency is smoke-tested in the mode model so abandon-ship cues render as critical alerts, destroyed selected-target cues render as neutral status, hostile lock/return-fire cues render as threat cues, local repair cues render as repair cues, and routine prompts render as next-action guidance in the bridge header, highlighted action buttons, cue tooltips, and matching recent-action log tags
- armed contacts holding engagement range build weapon-solution clocks with WEG range-band text
- ready hostile weapon-solution clocks emit hostile-fire opportunity events on live tactical ticks and can resolve automatic WEG gunnery/damage shots in the live overlay
- contact movement events now include combined engagement context: current sensor track confidence plus weapon-solution buildup/ready status
- selected-contact movement posture is model-formatted from the latest movement event when available, reporting tracking, closing, holding range, patrol movement, or movement blocks without exposing hidden-contact identity too early
- selected hostile contacts now show a model-formatted fire posture in the side panel, distinguishing acquiring, partial, ready-to-fire, armed, destroyed, and weapons-offline states from the raw WEG weapon-solution counter
- successful evasive piloting maneuvers can clear hostile weapon-solution clocks
- successful player gunnery hits can disrupt the target's weapon-solution clock, with the disruption callout formatted by the compact status model
- selected hostile contacts can require a ready weapon-solution clock before counterfiring
- gated counterfire reports how many weapon-solution rounds it consumed
- live traffic ticks combine contact movement with ship-condition countdowns for the player ship and contacts, then format movement, range holds, movement blocks, weapon-solution pressure, ready hostile-fire opportunities, automatic hostile shots, resulting player ship condition, and changed conditions through the compact status model
- live tactical telemetry reports traffic countdown, known and hidden contacts, authoritative current hostile lock pressure with track confidence from the latest movement event or persisted sensor confidence fallback, latest automatic hostile-fire summary, banked crew-station assists with their target actions, requested target aliases when they differ, and station-round age, player ship condition, system-damage flags, derived repair counts, crew-wound counts, and automatic update count
- maneuver hazards are drawn on the tactical map as visible approach zones with difficulty labels; the playable approach route now crosses both collision-capable debris and a non-collision traffic slot, so the maneuver button/header exercise multi-hazard difficulty and hazard-count cues from real data, while maneuver action readouts name both hazards in compact status text; clicking a hazard writes a status-model detail readout with piloting modifier, radius, collision-risk wording, and whether the current planned maneuver crosses or avoids that hazard, then logs it as a deliberate bridge action

This is not full space combat yet. It is the visual and data foundation for WEG-style piloting, sensors, gunnery, shields, astrogation, hull damage, scale, fire arcs, and crew-station play.

Design direction:

- WEG space rules lead.
- SW_MUSH space traffic and ship data are reference material, not a literal port target.
- MMO feel matters: the tactical plane should support readable shared encounters, not command-line-only procedures.
- Space gunnery uses the common scale helpers for attack, dodge, damage, and soak rather than bespoke starfighter/capital shortcuts.
- Crew-station assists are intentionally lightweight: the prototype rotates through data-driven bridge assists, banks one assist bonus in state, and consumes it on the next matching action. That keeps MMO crew play readable without recreating a turn-by-turn bridge console yet.
- Sensor-family crew assists deliberately accept natural data names such as `identify`, `identification`, `id`, `targeting`, `targeting_solution`, and `sensor_lock`, then bank them as sensors assists so contact identification and other sensors actions can consume them through one tested path.
- Commander tactical-coordination currently uses that same sensor-family path: the playable data includes a commander crew member and a `targeting_solution` assist that banks as sensors support for later contact work.
- Communications-family crew assists likewise accept natural data names such as `hail_contact`, `contact_hail`, `traffic_control`, `transmit`, `broadcast`, and `transponder_ping`, then bank them as communications assists so comms hail actions consume them through the same tested path.
- Engineering-family crew assists accept natural data names such as `engineering`, `damage_control_repair`, `field_repair`, `patching`, and `systems_repair`, then bank them as repair assists so WEG damage-control actions consume them through the same tested path.
- Damage-control data can set per-system repair difficulty overrides through `repair_difficulties` or `system_repair_difficulties` on either the ship/contact or current condition. Keys normalize like other repair systems (`sensor mast` and `sensor_mast` match), values may be WEG band names (`easy`, `moderate`, `difficult`, `very difficult`), numeric target difficulties, or `yard only`; condition-level entries override ship defaults. Destroyed weapons remain non-field-repairable even if data supplies an easier override. The selected-contact repair readout shows any override next to that repair target, such as `sensor_mast (Difficult)` or `backup_relay (Yard Only)`.
- The playable Mos Eisley approach data now demonstrates this on the visible Hutt Courier contact: its `cargo_lift` and `sensor_mast` repair targets display Very Difficult/Difficult labels in the selected-contact side panel and are covered by the data smoke test.
- Helm/piloting crew assists accept natural data names such as `helm`, `helm_control`, `evasive`, `break_lock`, `course_correction`, and `approach_vector`, then bank them as maneuver assists so WEG piloting and evasive break-lock actions consume them through the same tested path.
- Fire-control crew assists accept natural data names such as `fire_control`, `target_lock`, `weapon_lock`, `weapon_solution`, `bracketing`, `turret`, and `cannon`, then bank them as gunnery assists so WEG gunnery exchanges consume them through the same tested path.
- Shield-operation crew assists accept natural data names such as `shield_reroute`, `starship_shields`, `deflectors`, `deflector_shields`, `angle_shields`, and `shield_timing`, then bank them as shields assists so WEG starship-shields reroute actions consume them through the same tested path.
- Navigation-family crew assists accept natural data names such as `nav_plot`, `route_plot`, `jump_plot`, `jump_calculation`, `hyperdrive_calculation`, `hyperspace_route`, and `local_jump`, then bank them as astrogation assists so WEG route-plot actions consume them through the same tested path.
- Station-assist packets now preserve the requested target action alongside the normalized mechanics target through banking, replacement, and consumption. Status-model station action text, replacement callouts, consumed-assist suffixes, the bridge station strip, and live banked-assist telemetry show the normalized target with the original alias in parentheses when they differ; the bridge strip and telemetry also show the station round where the assist was banked, keeping bridge-support logs readable without weakening canonical consumption.
- The compact status model shows currently banked crew-station assists, including the action each bonus is reserved for and the station round when it was banked, so a later player action can visibly benefit from another crew member's successful WEG support roll.
- If a successful crew-station assist targets an action that already has a banked assist, the event records the replaced assist and the station readout reports the replacement. That keeps overlapping bridge support auditable without stacking unbounded bonus dice.
- Station assist suffixes, replacement callouts, shield-reroute action readouts, and station-assist action readouts are now status-model formatted and smoke-tested, so sensors, gunnery, shields, maneuver, repair, and crew-station actions all share the same bridge-support phrasing.
- Contact identification is available from the space overlay with `I`. It rolls sensors against an identification difficulty modified by the selected contact's current track quality, consumes any banked sensors assist, stores a curated transponder/affiliation/threat profile on success, and adds the ID suffix to contact labels. It is deliberately informational right now and does not alter hit math.
- Communications hail is available from the space overlay with `X`. It rolls the ship's communications pool against identity and threat pressure, consumes any banked communications assist, stores a responsive/unresponsive disposition, and displays a data-driven contact response. If the action data opts in, a successful hail can delay the selected contact's weapon-solution clock by a configured number of rounds, while a failed hail can advance an armed contact's clock as pressure and explicitly marks the lock ready if that pressure completes the firing solution; that lock context persists into the contact's comms disposition label/detail text, giving comms an encounter-pacing role without changing WEG gunnery hit math.
- Navigator astrogation plotting is available from the space overlay with `Y`. It rolls the ship's astrogation/navigator pool, consumes any banked astrogation assist, applies current hyperdrive calculation and astrogation difficulty penalties, clears those penalties on success, and fails without clearing state if the hyperdrive is disabled. This is route-prep/condition recovery, not hyperspace travel yet.
- Shield arcs now support front/rear/left/right coverage. Gunnery derives the incoming arc from target heading and relative attacker position unless a scripted `incoming_arc` or attacker `target_arc` override is present.
- Maneuver actions are still compact: they support turn and move intent plus WEG movement failure bands. Circular approach hazards can raise difficulty when crossed; collision-band failures become wild spins in open space or speed-based hull collisions when the scene marks an obstacle present.
- Maneuver hazard callouts are status-model formatted and smoke-tested, including quiet, zero-modifier, difficulty-modifier, and compact multi-hazard naming cases.
- Maneuver action readouts are now status-model formatted and smoke-tested end to end, including hazard, collision, wild-spin, broken weapon-solution, first crew-wound, consumed assist, and ship-condition callouts.
- Hazard visualization is intentionally simple: map-space circles and labels make the rules-visible danger zone legible before a pilot commits to the maneuver, and click detail keeps the WEG difficulty/collision consequence readable without requiring a maneuver roll first. The crossing/avoidance preview comes from the same tactical model route packet used by the maneuver roll, so UI warning text and WEG difficulty math stay aligned; the bridge header summarizes the route's total maneuver difficulty and names the first crossed hazard, while the maneuver button shows the current difficulty plus either a single-hazard marker or a multi-hazard count before the player commits. The current Mos Eisley approach data uses Approach Debris for collision risk and a Traffic Slot for non-collision traffic-control pressure.
- Contact movement now runs on a live five-second traffic clock with heading-based movement profiles, optional player-tracking behavior, engagement-range holds, weapon-solution clocks, explicit movement-block reasons for destroyed/drives-disabled/control-locked contacts, and condition countdowns. The clock still advances deterministic WEG accounting ticks under the hood, but the player no longer has to press a turn-advance key; `T` pauses/resumes the live clock for prototype inspection, and `;` advances one manual accounting tick for deliberate step-through testing.
- Live traffic tick readouts are now formatted by the compact status model and covered by smoke tests. They summarize moved contacts, range-holding contacts, weapon-solution readiness with sensor-track confidence, ready hostile-fire opportunities, automatic hostile shots, resulting player ship condition, and any condition countdown changes from the tick event packet.
- Engagement context combines persisted sensor confidence with weapon-solution counts on contact movement events. It is meant for server-authoritative telemetry and UI summaries, not as an extra difficulty modifier.
- The compact status model now consumes the latest movement-event engagement context for lock summaries, while preserving raw weapon-solution-count fallback text when no movement event is available. Current `weapon_solution_counts` remains authoritative, so consumed or cleared locks are hidden even if the previous movement event still described them. The fallback also uses persisted sensor confidence so a known hostile lock can still show track quality.
- Weapon-solution clocks are stateful encounter pressure that now bridge into real-time hostile fire. A hostile that keeps holding range can become ready after a configured number of tactical ticks; breaking the solution clears the count. Ready clocks emit explicit hostile-fire opportunity events, and the live overlay can resolve those opportunities through the same WEG gunnery, damage, shield, hull, system-effect, passenger-damage, crew-wound, and condition-persistence path used by player fire.
- The current player maneuver can act as an evasive break-lock action. It only clears weapon-solution clocks on a successful WEG piloting roll, so failed maneuvers leave the pressure intact and may still trigger normal movement-failure consequences.
- Player gunnery also has counterplay value: a successful hit against a contact with an active weapon-solution clock clears that target's count. Misses leave the lock pressure in place.
- Gunnery lock-disruption text is now status-model formatted and smoke-tested, keeping the hit-clears-lock feedback aligned with server-authoritative weapon-solution state.
- Counterfire can now be gated by that same clock. The Sensor Shadow requires a ready weapon solution before it can return fire, so the encounter rhythm is: acquire/hold solution, player responds with evasive maneuver or gunnery disruption, then unresolved ready locks can fire either as live-tick hostile shots or as counterfire after a player exchange.
- When gated counterfire or live-tick hostile fire resolves, it consumes the ready lock and reports the prior lock rounds. If the hostile is disabled/destroyed, or the player ship is already destroyed, by the time automatic fire would resolve, the blocked-fire event clears and reports the stale lock instead of preserving bogus pressure. If prior automatic fire in the same accounting tick destroys the player ship, later ready contacts block as prior-fire destruction and clear their locks. Automatic hostile-fire readouts include the resulting player ship condition, and the live telemetry line keeps a compact latest hostile-fire summary visible after the transient traffic message. Hostile fire deliberately does not consume player-banked station assists, keeping bridge support reserved for player-triggered actions.
- Sensor confidence now has a narrow mechanical role: it reports and persists how cleanly a contact was resolved from the WEG sensor roll margin, identification scans use it as their own difficulty modifier, and player gunnery against hidden or `sensor_targeting_required` contacts adds a targeting difficulty modifier by track quality: unresolved/missed +10, faint +6, partial +3, solid +1, clear +0. Visible/known targets remain normal, and hostile automatic fire/counterfire does not use the player ship's sensor picture as a penalty.
- The modal space mode now uses an opaque tactical presentation and suppresses ground controls, ground range pressure, ground HUD visibility, character-sheet visibility, and moving range-target animation while visible. This is a prototype stand-in for an actual mode transition: when the player is in space, the player object is no longer acting in the ground scene. The tactical layout is driven by a smoke-tested viewport layout model and rebuilds on viewport changes so the map and station panel keep filling the available play area. The modal view also lets the player click contact dots or labels to select the active space target. The selected contact is highlighted, shown in the side panel with model-formatted tactical details, range-aware weapon-solution lock state, WEG scale posture, defensive posture, hull/shield soak posture, weapon posture, crew wound posture, systems posture, bridge cue with action hotkeys, hostile-fire readiness, counterfire posture, destroyed-weapon-aware and custom data-preserving field-repair options, and the current sensor-targeting difficulty preview, and used by gunnery and target-side damage-control fallback; custom data-driven repairable systems can now be selected, repaired, cleared from the condition, and routed through fallback damage control without changing WEG roll math. Automatic field damage-control target routing skips systems marked `Yard Only`, while selected-contact labels still show those systems as damage worth inspecting or handling later in a bay.
- Keyboard target cycling is available with `Tab` or `.` for next contact and `,` for previous contact. The pure selection model is smoke-tested and includes hidden/unresolved returns so the player can drive the current prototype encounter even when a contact is not easy to click. Click and keyboard target changes share a smoke-tested status formatter, update the scan readout, and log as deliberate `Target` bridge actions.
- The recent-action log is deliberately compact and driven by a smoke-tested pure model, including one-shot cue-tag consumption for highlighted bridge actions. It keeps a short memory of player-triggered bridge actions across target selection, sensors, ID, comms, gunnery, shields, repair, astrogation, station assist, and maneuver actions so the player can follow the last few outcomes even when individual readout panels are overwritten.
- Gunnery action readouts now come from the compact status model and are smoke-tested end to end, including hit/miss, damage/soak pools, shield contribution, system effects, passenger/crew wound callouts, consumed assists, targeting context, ship condition, lock disruption, and counterfire.
- Sensor sweep readouts are formatted by the compact status model and covered by smoke tests. They include successful tracks, newly revealed contacts, consumed sensor assists, and compact known/hidden contact totals immediately after the sweep result, so the player can see whether the latest WEG sensors action changed the tactical picture without waiting for the live traffic telemetry line.
- Contact map labels are also status-model formatted and smoke-tested, including unresolved hidden returns, revealed contact status, sensor-track confidence suffixes, successful identification suffixes, and communications disposition suffixes.
- Starship condition persistence currently tracks shield loss, controls ionized, hull severity counts, system effects, lost Move, disabled weapons, disabled drives, hyperdrive flags, structural damage, passenger/gunner damage logs, crew wounds, repair logs, and destruction. Damage control can now repair targeted condition groups and reports free field repair time plus dockyard cost estimates. Astrogation plotting can clear hyperdrive calculation and astrogation difficulty penalties after successful route prep. In the live overlay it prioritizes the local ship after counterfire/collision damage, then falls back to the drill target if the player ship is clean. The overlay now presents compact condition summaries after gunnery, repair, astrogation, and maneuver actions. Richer station-by-station recovery is next.
- Damage-control event packets now include before/after condition summaries, and the live repair readout is status-model formatted and smoke-tested with compact condition changes, WEG difficulty names beside the target number, yard-only field-repair-unavailable wording, repair quotes, consumed assists, and final ship condition text. Damage-control target routing skips destroyed weapons and `Yard Only` systems, then falls through to the next field-repairable system, matching selected-target repair labels and telemetry. Banked station assists now show the contributing crew action and bonus dice when consumed by sensors, gunnery, shields, maneuver, or repair actions.
- Compact space status treats persistent `worst_hull_severity` and destroyed flags as authoritative for the hull label, so telemetry matches the starship damage state written by gunnery, counterfire, and collision events.
- Shared ship-condition action text is now status-model formatted and smoke-tested for hull severity, shields, controls, hyperdrive/astrogation flags, and crew-wound counts. Gunnery, damage-control, maneuver, and counterfire readouts use that same formatter.
- Compact space status also counts persisted crew wounds with severity above stun-only results, making passenger/gunner damage visible in the live MMO-style telemetry line.
- Compact space status now surfaces persistent system-damage flags such as maneuverability loss, lost Move, disabled weapons, disabled drives, disabled hyperdrive, generator overloading, and structural damage. These are read-only telemetry for condition state already produced by WEG starship damage results.
- Compact space status derives repair counts from the same persistent fields damage control can target, while excluding destroyed weapons and `Yard Only` systems from field-repairable counts. Selected-contact repair labels still show yard-only damage so players can see what needs dockyard work.
- Counterfire currently reuses the same WEG gunnery primitive as player fire: an armed, non-destroyed hostile resolves a return shot with its own gunnery, fire-control, damage, shield-arc, and condition effects, then persists damage on the player ship. Its readout is now status-model formatted and smoke-tested, including blocked counterfire reasons, spent weapon-solution locks, hit/miss results, damage names, and attacker condition summaries.
- Repair cost notes are currently metadata only. Field damage control stays free as a skill moment, while dockyard quotes use SW_MUSH yard-repair percentages with GG6 flat damaged-ship costs as fallback.
